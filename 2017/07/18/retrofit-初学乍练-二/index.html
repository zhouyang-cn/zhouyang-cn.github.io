retrofit-初学乍练(二)

发表于 2017-07-18 | | 阅读次数

上一篇文章，用retrofit请求github的api写了一个简单的demo，本文将继续学习retrofit的基本用法。

请求方法

Http请求方式不太了解的童鞋可以看一下这篇文章，retrofit的GET方式我前面已经尝试过了，下面要学习一下其他的请求方式。

当然retrofit针对http的所有请求方式都有对应的注解：@GET, @POST, @PUT, @DELETE, @PATCH or @HEAD.

一般情况下，我们会给这些注解传递一个相对路径的URL(eg：@POST("/tasks"))，而不是一个全路径的URL(eg：https://developer.github.com/v3/api/user/info)。

这样做的优点是：我们只需要设置请求的base Url，就可以在想要更换api的域名时，直接更换baseUrl就可以了，不需要进行很大的改动。

路径(Endpoints)详解

上一篇文章讲了一下retrofit的简单用法，我们定义了一个接口，将我们要请求的api定义成了一个个方法到这个接口里，那么下面学习一下这些方法具体规则：

以Call<Task> createTask(@Body Task task);为例，它包含一下三个部分：

  1. 方法名createTask
  2. 方法返回值Call<Task>
  3. 方法参数@Body Task task

首先，来看方法名，retrofit对方法名是没有特殊要求的，但是我们应该尽量使用可读性高的名字作为方法名；

其次，方法的返回值，这里retrofit是有要求的，我们的返回值必须是Call<>类型，而泛型中传递的才是我们实际需要的数据类型，比如，我们要从服务端获取用户信息，我们应该在泛型中传递一个UserInfo类型；假如，我们想获取原始的响应，泛型应该使用ResponseBody来代替；假如，我们不关心服务器返回的数据，我们可以给泛型传递Void。

最后，方法参数，retrofit定了很多注解方便我们添加参数：

  • @Body：将一个java对象作为请求参数发送请求
  • @Url：使用动态URL
  • @Path：路径参数
  • @Query：查询参数
  • ……
